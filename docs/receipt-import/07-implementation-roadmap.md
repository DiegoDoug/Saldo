### Stage 1 â€” Backend skeleton, no AI yet âś… done
`receipt_import` module scaffolded: `models.py` + migration + `core/metadata.py` registration, `storage.py` (content-addressed disk storage, `SALDO_RECEIPT_STORAGE_DIR`), `schemas.py`, and a `router.py` with upload/get/list/draft-patch/confirm/discard/image endpoints wired to a **stub pipeline** that runs inline and marks status `ready` with a hand-built fixed `DraftReceiptAnalysis` (no real OCR/AI calls, no `BackgroundTasks` yet â€” both land in Stage 2 once there's real latency to hide). Config additions: `SALDO_RECEIPT_STORAGE_DIR`, `SALDO_RECEIPT_MAX_UPLOAD_MB`, `SALDO_DEEPSEEK_API_KEY`/`_MODEL`/`_BASE_URL` (unused by the stub, wired now so the 503-when-unconfigured gate is real from day one). 9 new pytest cases covering upload validation (content-type/size), ownership scoping, content-hash dedup, the draft-patch/confirm/discard state machine (including the 409s for editing/confirming a non-`ready` receipt), and the disabled-feature 503. Full existing suite (85 tests) + `ruff check .` stay green.
**Why first:** de-risks the genuinely novel part of this feature (file upload, storage, a new table) independently of the AI integration, which is the part most likely to need iteration once tested against real receipts.
**Found and fixed during implementation:** the image-download endpoint would have thrown an unhandled `FileNotFoundError` (â†’ 500) for a discarded receipt, since `discard_receipt` deletes the on-disk file but keeps the row; it now returns a clean 404 instead.
**Trade-off:** the DeepSeek config fields exist a stage early, purely to make the 503 gate testable now instead of retrofitting it in Stage 2.
**Risk carried forward:** none new â€” this stage touches no existing table or router logic, only adds one net-new table and one net-new router.

### Stage 2 â€” OCR + DeepSeek pipeline âś… done
`ocr/` (`OcrProvider` protocol + `TesseractOcrProvider`, the decided default) and `ai/` (`ReceiptExtractionProvider` protocol + `RawExtraction`/`ExtractionContext` schemas + `DeepSeekProvider` + `ai/prompts.py`) provider abstractions; `extraction_service.build_context` (pulls the user's own categories/recent merchants into the prompt so semantic matching rides the one extraction call, per Document 5 Â§5); `draft_builder.py` mapping a `RawExtraction` into the `DraftReceiptAnalysis` contract (no real merchant/category *matching* yet â€” that's Stage 3, so today every match is exactly what the AI itself proposed); real `pipeline.py` replacing the Stage 1 stub, now scheduled via `BackgroundTasks` from `router.py` instead of awaited inline.
**Why second:** depends on Stage 1's storage/table; is the highest-uncertainty stage (prompt quality against real receipts), so it's isolated rather than bundled with matching logic.
**Found and fixed during implementation:** confirmed empirically (not from memory) that this project's pinned FastAPI version closes yield-dependencies â€” including the request's DB session â€” *before* running `BackgroundTasks`, not after. `pipeline.py` therefore opens its own session via `app.core.db.async_session_maker` rather than reusing the request's, and `conftest.py`'s `client` fixture now monkeypatches that same attribute to the ephemeral test database so background-task tests aren't silently working against the wrong (real, unmigrated) database. OCR/AI provider selection stays testable through ordinary `app.dependency_overrides` because `router.py` resolves both via `Depends` and hands the resolved instances into the background task, rather than the task re-resolving them itself.
**Trade-off:** dropped the `SALDO_AI_PROVIDER`/`SALDO_OCR_PROVIDER` config-driven dispatch this document originally sketched â€” with exactly one implementation of each Protocol, a `match` statement with one real branch was dead weight (`CLAUDE.md`'s "don't design for hypothetical future requirements"). The Protocols themselves already give a second provider a clean landing spot; a selector setting is one `if` away whenever that day comes.
**Also shipped:** Tesseract's system dependency wired everywhere it needed to be â€” `Dockerfile` (`tesseract-ocr` + `tesseract-ocr-spa`), CI (`.github/workflows/ci.yml`), and README's local-dev prerequisites â€” plus `.env.example` documenting every new `SALDO_DEEPSEEK_*`/`SALDO_RECEIPT_*`/`SALDO_OCR_*` variable. 12 new tests (5 pure `draft_builder` cases, a prompt-content check, 2 `DeepSeekProvider` tests with `httpx.AsyncClient.post` mocked exactly like `test_email.py`'s pattern, one real-Tesseract smoke test against a Pillow-rendered image, one `build_context` user-scoping test, plus the API-level pipeline success/category-and-merchant-match/failure tests) â€” full suite now 106 tests, `ruff check .` clean.
**Risk carried forward:** the DeepSeek prompt (`ai/prompts.py`) is untested against *real* receipt photos/OCR noise â€” only a hand-written few-shot example and a mocked response shape. Prompt quality against real-world receipts is the risk this stage's design doc flagged as highest-uncertainty, and it remains unresolved until it's tried against real DeepSeek calls (needs a real `SALDO_DEEPSEEK_API_KEY`, deliberately not exercised in CI).

### Stage 3 â€” Merchant & category matching âś… done
`merchant_matching.py` (`normalize_merchant_name` + exact/fuzzy tiers via `rapidfuzz`, falling through to the AI's own semantic guess, then to "no match") and `category_matching.py` (merchant-default â†’ existing-similarity â†’ AI-semantic â†’ suggest-new, income categories excluded since a receipt is never income), wired into `pipeline.py` ahead of `draft_builder.build`, which now takes the already-computed `MerchantMatch`/`CategoryMatch` as arguments instead of deriving them from `RawExtraction` itself (Stage 2's stand-in logic moved out of `draft_builder.py` entirely, not layered alongside it).
**Why third:** depended on Stage 2's `RawExtraction` shape; is the part of the brief (Phases 4â€“5) with the most explicit behavioral requirements ("never create duplicate merchants when confidence is high"), so it got isolated review attention.
**Found and fixed during implementation:** the first version of the end-to-end "semantic match" test was silently invalid â€” `rapidfuzz.fuzz.WRatio("amazon", "amzn mktp us")` scores 72, above the fuzzy-match threshold, so a merchant named "Amazon" would have matched at tier 2 (fuzzy) rather than exercising tier 3 (semantic) as the test claimed to prove. Caught by actually running `rapidfuzz` against the candidate strings before trusting the test, not by inspection; swapped in "Netflix" (score ~26 against the same OCR text) to genuinely force the fallback path.
**Trade-off:** dropped a `FUZZY_AUTO_THRESHOLD` constant this stage's implementation initially introduced â€” confidence already scales continuously with the match score, so a separate "auto vs. suggest" band was a second knob doing the same job the existing confidence-threshold UI convention (Document 2 Â§7) already does. One threshold (`FUZZY_SUGGEST_THRESHOLD`, the floor for returning a match at all) is enough.
**Also shipped:** `merchant_matching.match`/`category_matching.match` and `draft_builder.build` are pure/near-pure functions taking a session + plain values, so most of Stage 3's 16 new tests (4 draft_builder, 5 merchant_matching, 5 category_matching) need no HTTP layer at all â€” plus 2 new API-level tests proving the full priority chain end-to-end (exact merchant match winning over an absent AI guess and feeding a merchant-default category; a genuinely non-fuzzy-matching pair falling through to the AI's semantic guesses for both merchant and category). Full suite now 152 tests, `ruff check .` clean.
**Risk carried forward:** fuzzy-matching thresholds (`FUZZY_SUGGEST_THRESHOLD = 65` for merchants, `70` for categories) are reasonable starting points, not tuned against real user data â€” a self-hoster with many similarly-named merchants (e.g. multiple corner stores) may see more "did you mean" prompts than ideal, or occasional wrong auto-matches. Worth revisiting once Stage 5 (frontend review screen) makes these tradeoffs visible to a real user.

### Stage 4 â€” Frontend: capture, upload, polling âś… done
`ReceiptDropZone.tsx` (file input + drag/drop, client-side type/size pre-check), `apiUploadRequest` (the client's first multipart addition), `receipt-import/api.ts` (snake_caseâ†”camelCase mapping for the whole `DraftReceiptAnalysis` contract, typed now even though Stage 4 only renders a summary of it), `hooks.ts` (`useUploadReceipt`/`useReceiptImport` polling/`useDiscardReceipt`), `useOnline.ts`, and `ReceiptImportDialog.tsx` reusing `ForgotPasswordDialog`'s modal shell â€” the "Escanear recibo" entry point sits next to "+ Nuevo" on `TransactionsPage.tsx`, disabled offline. The "ready" step renders a real (if minimal) summary â€” merchant/amount/date/confidence plus a collapsible raw-JSON pane â€” not a bare placeholder; editing and confirming are still Stage 5.
**Why fourth:** the first frontend stage, deliberately scoped to stop before the most complex UI (review form) so upload/polling plumbing is validated in isolation.
**Found and fixed during implementation:** two real bugs, both caught by actually running the tests/browser rather than by inspection. (1) `userEvent.upload` silently filters files against the input's `accept="image/*"` before they ever reach application code â€” a test asserting the "unsupported file type" rejection via the file input was vacuously passing for the wrong reason; switched it to a real `fireEvent.drop` (which a real browser lets bypass `accept` entirely) to actually exercise that code path. (2) TanStack Query v5 calls a mutation function with a second (context) argument, so `expect(fn).toHaveBeenCalledWith("id")` never matches; a test spent a while looking like a timing flake before printing the actual call arguments showed the real shape. Both are now correctly asserted (`expect.anything()` for the second arg; `fireEvent.drop` for the type-rejection path).
**Trade-off, explicit deviation from Document 6:** the "hide the button when the feature is unconfigured" capability check from Document 6 Â§1/Â§3 was **not** built â€” there's no boot-time config endpoint to piggyback on yet, and adding one just for this one flag would be exactly the premature abstraction `CLAUDE.md` warns against. Instead the button always renders (only gated on having an account and being online) and a failed upload surfaces the backend's 503 inline ("El escaneo de recibos no estĂˇ configurado en este servidor."). Functionally equivalent for a self-hoster who hasn't set a key â€” they see one clear message on first attempt instead of a hidden button â€” but worth revisiting if a real settings/capabilities endpoint appears for other reasons.
**Manually verified end-to-end in a real browser** (Playwright driving the actual dev server + backend, per this session's UI-verification standard) against three real scenarios, not mocked: registered a user, created an account, opened the dialog from `TransactionsPage`, and (a) with no `SALDO_DEEPSEEK_API_KEY` set, uploaded a real image and confirmed the inline 503 message and graceful return to the capture step; (b) with a fake (non-working) key set, uploaded a real image and watched it flow through the *real* pipeline â€” actual Tesseract OCR ran, a real HTTP call went to `api.deepseek.com`, got a real `403 Forbidden`, and the dialog correctly landed on the failed state with working Reintentar/Descartar buttons. Screenshot confirms the button placement and dialog styling match the rest of the app.
**Verified:** 8 new component tests (`ReceiptDropZone`: valid/oversized/wrong-type; `ReceiptImportDialog`: capture/ready/failed/503/discard, api.ts mocked via `vi.mock`) plus the existing 106, full suite 114 tests, `tsc --noEmit` and `vite build` both clean.
**Risk carried forward:** the 503-inline-message approach means a disabled feature is only discovered by trying it, not upfront â€” acceptable for now (Stage 4 scope), revisit if user feedback says otherwise.

### Stage 5 â€” Frontend: review, confirm, inline create âś… done
`ReceiptReviewForm.tsx` (per-field confidence badges â€” 0.9 threshold for amount/date, 0.7 elsewhere â€” plus a warnings banner), `EntityCombobox.tsx` (the merchant/category inline-create combobox generalized from `TagInput`), and the confirm flow: writes the Transaction to Dexie via the existing, unmodified `transactions/localRepo.ts`, fires `POST /receipt-imports/{id}/confirm` best-effort (the transaction write is what matters; a failed history-link call isn't worth blocking the user over), then shows a `ForgotPasswordDialog`-style success step.
**Why last:** the highest-stakes stage from a data-correctness standpoint (this is where a wrong AI suggestion could become a real transaction if the review UX isn't clear) â€” reviewed last, with everything underneath it already proven.
**Deviation from Document 6:** inline merchant/category creation calls the local Dexie `localRepo` functions (`addMerchant`, `addCategory`) directly rather than Document 6 Â§2's original sketch of a direct `POST /merchants`/`POST /budgeting/categories` call mirrored into Dexie afterward. Simpler, and keeps the whole confirm flow offline-first-consistent â€” the new row syncs up on the next push exactly like the transaction itself, with no two-write-paths-to-keep-in-sync risk. `budgeting/localRepo.ts`'s `addCategory` was changed to return the new id (was `Promise<void>`) since nothing needs the id at creation elsewhere.
**Found and fixed during implementation â€” a real, non-obvious bug, caught only by testing in a real browser:** `EntityCombobox`'s initial displayed text was set once via `useState(selected?.name ?? "")` at mount. `useAccounts`/`useCategories`/`useMerchants` are Dexie `useLiveQuery` hooks that resolve *asynchronously* â€” so on first render, before they've loaded, `selected` is `null` even when the draft already carries a real matched id, and the field would silently render blank forever (the `useState` initializer never re-runs). Every automated test had mocked those hooks to return data *synchronously*, which hid the bug completely â€” 20 passing component tests gave no signal here. Manually driving the actual app against a real backend (with a small local stub standing in for DeepSeek's API, so the full real pipeline â€” real Tesseract OCR, real merchant/category matching against real synced data â€” could run without a paid API key) surfaced blank Comercio/CategorĂ­a fields immediately. Fixed with a `useEffect` that re-syncs the display text once the matching option actually arrives, without clobbering in-progress typing; added a regression test (`EntityCombobox.test.tsx`, "syncs the display text once a matching option arrives after mount") that renders with empty options then rerenders with the real one, exactly reproducing the timing that unit tests with synchronous mocks couldn't catch. Also added an `initialQuery` prop so a *suggested-but-unmatched* name (no id at all) still shows up as editable text instead of a blank field.
**Manually verified end-to-end in a real browser**, real pipeline, no mocks: registered a user, created an account (which seeds default categories, including "Supermercado"), uploaded a real image against a backend pointed at a local stub DeepSeek server returning a realistic extraction. Watched the review form correctly prefill from a real `existing_similarity` category match (the seeded default "Supermercado" category matched the stub's suggested category name for real, via `category_matching.py`'s actual fuzzy tier) and a merchant with no match (correctly offering "Crear Â«MercadonaÂ»"), confirmed the date's sub-threshold confidence showed a "Revisar" badge while amount's didn't, created the merchant inline, clicked Confirmar, saw the success screen, and watched the new transaction appear live in the list behind the dialog (Dexie's `useLiveQuery` re-rendering instantly, no reload) â€” screenshots on file. Did not verify the cross-device sync round-trip (a second session pulling the confirmed transaction) â€” that exercises only the pre-existing, already-proven sync engine with data from a new source, not new code, so it was judged lower-value than the bug this same pass caught.
**Verified:** 6 new tests (`EntityCombobox`: select/create/no-duplicate-create/clear + the two async-timing regression tests) and 5 new tests (`ReceiptReviewForm`: prefill, confidence flagging, inline create, confirm writes the right transaction, discard), plus `ReceiptImportDialog.test.tsx` updated for the new review-form-based ready step. Full frontend suite 126 tests, `tsc --noEmit` and `vite build` both clean. Backend untouched this stage (152 tests, unchanged).
**Risk carried forward:** the confirm-link call (`POST /receipt-imports/{id}/confirm`) is fire-and-forget from the UI's perspective â€” if it fails, the transaction is still safely created but that receipt's history entry never gets `linked_transaction_id` set. Acceptable (nothing user-visible breaks, and history browsing isn't built yet â€” Document 2 Â§7), but worth a retry/backfill mechanism if a future history UI depends on that link being reliably present.

## Standing rules for every stage (inherited from `docs/transformation/06`, and this session's own instructions)

- Tests written and passing (`pytest`, `vitest`, `tsc --noEmit`), `ruff check .` clean, before a stage is presented as done.
- The security invariant â€” every `receipt_import` query filtered by `user.id` â€” is checked explicitly in each stage's own review, not assumed from Stage 1.
- No stage touches `syncEngine.ts`, the sync router, or the Dexie version number for `ReceiptImport` itself (Document 4) â€” if a later stage seems to need that, treat it as a signal the design has drifted and stop to reconsider rather than pushing through.
- At the end of each stage: architectural decisions taken, trade-offs, and risks are stated explicitly, and the next stage waits for approval â€” exactly as instructed.
